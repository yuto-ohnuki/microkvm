# Step 4: プロテクトモード → ロングモード (64-bit)

## 目的

ゲスト CPU を 32-bit プロテクトモードから 64-bit ロングモードに遷移させる。
**ページテーブル**、**CR3**、**CR4.PAE**、**EFER.LME** を導入し、ハードウェア仮想化の中核である2段階アドレス変換を明らかにする。

## 背景

### ロングモードの要件

64-bit ロングモードに入るには、4つの条件を順番に満たす必要がある:

1. **ページテーブル**をメモリに設定し、CR3 で指す
2. **CR4.PAE = 1** — ページテーブルエントリを 64 ビットに拡張
3. **EFER.LME = 1** (MSR 0xC0000080, bit 8) — ロングモード使用の意思を宣言
4. **CR0.PG = 1** — ページングを有効化

PAE と LME が設定された状態でページングを有効にすると、プロセッサはロングモードを起動する (EFER.LMA が内部的にセットされる)。ただし 64-bit 命令デコードはまだ始まらない — CPU は**互換モード** (32-bit コードを実行するロングモードのサブモード) にある。その後の **L=1** のコードセグメントへの far jump が真の 64-bit 命令実行を開始する。

### x86-64 ページテーブル階層

ロングモードは通常4段階の階層を使用する:

```
CR3 → PML4 → PDPT → PD → PT → 4KB ページ
```

このステップでは 2MB ページ (PD エントリの PS=1) を使ってウォークを早期終了するため、PT レベルはスキップされる:

```
CR3 → PML4 (Page Map Level 4)
        └→ PDPT (Page Directory Pointer Table)
              └→ PD (Page Directory)
                    └→ 2MB ページ (PS=1, ここで変換終了)
```

各エントリは 8 バイト。テーブルは 4KB アラインのため、下位 12 ビットはフラグに使える:

| Bit | フラグ | 意味 |
|-----|--------|------|
| 0 | P | Present — エントリが有効 |
| 1 | RW | 書き込み可能 |
| 7 | PS | Page Size — このレベルでウォークを終了 |

### アイデンティティマッピング

仮想アドレス 0 → 物理アドレス 0 にマップする (アイデンティティマップ)。これは必須: コードは既に低アドレスで実行中なので、ページング有効化後も同じアドレスが同じ物理位置に解決される必要がある。アイデンティティマッピングなしでは、CPU は存在しない変換を通じて次の命令をフェッチしようとし、即座にページフォルトを引き起こす。

```
仮想 0x000000–0x1FFFFF  →  物理 0x000000–0x1FFFFF  (最初の 2MB)
```

### 2段階アドレス変換

ゲスト内でページングが有効になると、全てのメモリアクセスで2つの独立した変換が起きる:

```
ゲスト仮想アドレス
        │
        │  ゲストページテーブル (ゲストが構築, CR3)
        ▼
ゲスト物理アドレス
        │
        │  第2段階変換 (KVM が構築)
        ▼
ホスト物理アドレス
```

| | ゲストページテーブル | 第2段階 (EPT/NPT) |
|---|---|---|
| 構築者 | ゲストソフトウェア (OS, ブートローダー, またはゲスト代理の VMM) | KVM カーネルモジュール |
| 変換内容 | ゲスト仮想 → ゲスト物理 | ゲスト物理 → ホスト物理 |
| 格納場所 | ゲストメモリ (CR3 がここを指す) | KVM 内部構造体 |
| フォーマット | x86 ページテーブル | ハードウェア定義のネストページテーブル (EPT/NPT) |

ハードウェアはアドレス解決の一部として両方の変換を透過的に実行する。これがハードウェア仮想化がページング有効のゲストを全メモリ操作をトラップせずに実行できる理由。

この分離により、KVM は独自の「物理」メモリレイアウトを持つ仮想マシンを提示しつつ、任意のホストメモリでバッキングできる。以降のステップはこの上に構築される: MMIO ホール (Step 5)、dirty page tracking、スナップショット、live migration は全て第2段階変換レイヤーで動作する。

### なぜ VMM が EFER.LME を設定するか

実際のブートフローでは、カーネルが `wrmsr` を実行して EFER.LME を設定する。
ゲストをページングとモード遷移に集中させるため、vCPU セットアップの一部として VMM から EFER.LME を初期化する。QEMU が初期 vCPU 状態を準備する際と同じアプローチ。

## 実行フロー

```
VMM (microkvm.c)                         Guest (guest.S)
────────────────                         ───────────────
ゲストメモリにページテーブルを書く:
  0x70000: PML4[0] → 0x71003 (P+RW)
  0x71000: PDPT[0] → 0x72003 (P+RW)
  0x72000: PD[0]   → 0x00083 (P+RW+PS)

KVM_SET_SREGS: efer |= LME
                                         [リアルモード]
                                           'R' via PIO
                                           lgdt, CR0.PE=1, far jump
                                         [プロテクトモード]
                                           'P' via PIO
                                           mov $0x70000, %cr3
                                           CR4.PAE = 1
                                           CR0.PG = 1  ← ロングモード (互換)
                                           far jump (selector 0x18, L=1)
                                         [ロングモード, 64-bit]
                                           'L' via PIO
                                           hlt
```

Step 3 と同様に、モード遷移は完全にゲスト内で起きる。VMM は文字出力の PIO exit と最後の HLT しか見ない。

## 実装

### VMM: ページテーブル設定

```c
/* ページテーブルを GPA 0x70000 に配置 (GPA 0 のゲストコードより十分上) */
uint64_t *pml4 = (uint64_t *)((char *)mem + 0x70000);
uint64_t *pdpt = (uint64_t *)((char *)mem + 0x71000);
uint64_t *pd   = (uint64_t *)((char *)mem + 0x72000);

pml4[0] = 0x71000 | 0x3;   /* present + writable, PDPT を指す */
pdpt[0] = 0x72000 | 0x3;   /* present + writable, PD を指す */
pd[0]   = 0x0     | 0x83;  /* present + writable + PS (物理 0 の 2MB huge page) */
```

各エントリは物理アドレス (上位ビット) とフラグ (下位 12 ビット) を組み合わせる。アドレスは 4KB アラインである必要があるため、下位ビットはフラグに使える。

VMM がページテーブルを構築する理由 (ゲストではなく): ゲストバイナリは GPA 0 にロードされたフラットバイナリ。動的メモリアロケータがない。VMM が実行開始前にゲストメモリの既知の場所にテーブルを書く。実際のブートローダーや OS カーネルは自身のページテーブルを構築する。

### VMM: EFER.LME

```c
sregs.efer |= (1 << 8);   /* LME = Long Mode Enable */
ioctl(vcpufd, KVM_SET_SREGS, &sregs);
```

### ゲスト: 遷移シーケンス (プロテクトモード内)

```asm
    /* CR3 をページテーブルルートに設定 */
    mov $0x70000, %eax
    mov %eax, %cr3

    /* PAE を有効化 — ロングモードの前提条件 */
    mov %cr4, %eax
    or $0x20, %eax          /* bit 5 = PAE */
    mov %eax, %cr4

    /* ページングを有効化 → ロングモード起動 (EFER.LME は設定済み) */
    mov %cr0, %eax
    or $0x80000000, %eax    /* bit 31 = PG */
    mov %eax, %cr0

    /* 64-bit コードセグメントへの far jump (GDT[3], selector 0x18) */
    ljmpl *far_ptr
```

`CR0.PG = 1` の後、CPU はロングモード互換サブモード (まだ 32-bit コードを実行)。far jump で L=1 のディスクリプタを CS にロードすることで、CPU が 64-bit 命令デコードに切り替わる。

### ゲスト: 64-bit コード (手動エンコード)

GNU `as` は 32-bit モード (`--32`) で動作しているため、64-bit アセンブリを直接出力できない。これらの命令は両モードで同一のエンコーディングを持つため、ツールチェーンを切り替えずにロングモード実行を示せる:

```asm
long_mode:
    .byte 0xB0, 'L'     /* mov al, 'L' */
    .byte 0xE6, 0x10    /* out 0x10, al */
    .byte 0xB0, '\n'    /* mov al, '\n' */
    .byte 0xE6, 0x10    /* out 0x10, al */
    .byte 0xF4          /* hlt */
```

### GDT: 64-bit コードセグメント

```asm
    .quad 0x00209A0000000000    /* GDT[3] (selector 0x18): 64-bit コード */
    .quad 0x0000920000000000    /* GDT[4] (selector 0x20): 64-bit データ */
```

Step 3 の 32-bit ディスクリプタとの主な違い:
- **L=1** — CPU にこれが 64-bit コードセグメントであることを伝える
- **D=0** — L=1 の場合に必要 (D と L は同時に 1 にできない)
- Base と limit はロングモードでは無視される

## 出力

```
$ ./microkvm
Loaded guest: 4136 bytes
Starting guest...
R
P
L
Guest halted.
```

3文字が全3つの CPU モードの成功実行を確認: リアル → プロテクト → ロング。

## 重要な知見

このステップで初めてゲストが自身の仮想メモリを管理する。
以降、ゲストは物理アドレスを直接使って実行しない。全ての命令フェッチとメモリアクセスがゲスト制御のページテーブルを経由する。

ゲストは第2変換レイヤーの存在を知らない。アドレス 0 から始まる物理メモリを自分が所有していると信じている。ハードウェアが KVM の管理するネストページテーブルを通じて、ゲスト物理アドレスを実際のホストメモリに透過的に変換する。

この2段階モデルが以降の全ての基盤:
- **Step 5**: ゲスト物理空間の MMIO ホールがデバイスエミュレーションの exit をトリガー
- **Step 15–18**: dirty page tracking と live migration が第2段階で動作
- **Step 12–14**: virtio がゼロ exit データ転送に共有ゲスト物理メモリを使用

## このステップで学べること

| 概念 | ここでの現れ方 |
|------|--------------|
| ページテーブル | 2MB huge page による 3段階アイデンティティマップ (PT レベルはスキップ) |
| CR3 | PML4 を指す — ゲストアドレス変換のルート |
| CR4.PAE | ロングモードの前提条件 (64-bit ページテーブルエントリ) |
| EFER.LME + LMA | LME が意思を宣言; LMA は PG 設定時に有効化 |
| 互換モード | PG=1 後だが far jump 前 — まだ 32-bit 実行 |
| 2段階変換 | ゲスト PT + 第2段階 (EPT/NPT) — ハードウェアが両方をウォーク |
| アイデンティティマッピング | ページング有効化の瞬間をコードが生き残るために必要 |
| Far jump (L=1) | 64-bit ディスクリプタで CS をリロードし 64-bit 実行を開始 |

### Linux カーネルとの対応

Linux カーネルは初期ブート中に同じシーケンスを実行する:
1. 一時的なアイデンティティマップページテーブルを構築
2. CR3 をロード
3. CR4.PAE を有効化
4. EFER.LME を設定
5. CR0.PG を有効化
6. 64-bit コードへの far jump

microkvm は同じアーキテクチャ要件に従うが、最小のページテーブル設定を使用 (PML4 1つ、PDPT 1つ、2MB huge page の PD 1つ)。
