# Step 11: Interactive shell

## Goal

Add serial **RX** (receive) support so the guest can accept input from the host terminal. Combined with a busybox initramfs, this gives us an interactive shell running inside the VM - completing the "usable VMM" milestone.

## Background

### TX vs RX

Step 10 implemented serial TX (transmit): the guest writes to THR, the VMM outputs to stdout.
Step 11 adds the reverse direction:

| Direction | UART role | Flow |
|-----------|-----------|------|
| TX (Step 10) | Guest → Host | Guest writes THR → VMM prints to stdout |
| RX (Step 11) | Host → Guest | User types → VMM sets RBR → IRQ 4 → Guest reads |

### Why a separate stdin thread?

The vCPU thread is blocked inside `ioctl(KVM_RUN)` most of the time. It cannot simultaneously wait for host keyboard input. A dedicated **stdin reader thread** solves this:

```
[Host terminal]  →  [stdin_thread]  →  [UART RBR + IRQ 4]  →  [Guest kernel]
   keystroke         read(stdin)        uart_rx()               serial driver
```

This is conceptually similar to QEMU's character device backends, which handle I/O independently of vCPU execution.

### UART RX registers

| Register | Role |
|----------|------|
| RBR (0x3F8, read, DLAB=0) | Receive Buffer - guest reads the character here |
| LSR bit 0 (DR) | Data Ready - "there is unread data in RBR" |
| IER bit 0 (RDI) | Receive Data Interrupt enable |
| IIR = 0x04 | Interrupt identification: "cause is received data" |

### RX sequence

```
1. User presses 'a'
   stdin_thread: read(STDIN_FILENO) returns 'a'

2. stdin_thread updates UART state:
   uart.rbr = 'a'
   uart.lsr |= DR (Data Ready)

3. stdin_thread injects IRQ 4 (if IER.RDI is enabled):
   KVM_IRQ_LINE {irq=4, level=1}  (assert)
   KVM_IRQ_LINE {irq=4, level=0}  (deassert)

4. Guest kernel receives IRQ 4:
   serial8250_interrupt() runs

5. Guest reads IIR (0x3FA) → VMM returns 0x04 (RDI)
   Driver knows: "interrupt is due to received data"

6. Guest reads RBR (0x3F8) → VMM returns 'a'
   VMM clears: lsr &= ~DR, pending_rdi = 0

7. Driver passes 'a' to tty layer → shell processes it
   Shell echoes 'a' back via TX → appears on host terminal
```

### Raw terminal mode

By default, the host terminal buffers input line-by-line (canonical mode).
For an interactive shell, we need each keystroke delivered immediately:

```c
struct termios raw = orig_termios;
raw.c_lflag &= ~(ICANON | ECHO);  /* no line buffering, no local echo */
raw.c_cc[VMIN] = 1;                /* read returns after 1 byte */
tcsetattr(STDIN_FILENO, TCSANOW, &raw);
```

The terminal is restored on exit via `atexit(restore_terminal)`.

### ESC sequence filtering

The kernel's serial driver queries cursor position (`ESC[6n`) during init.
The host terminal responds with `ESC[row;colR`. Without filtering, this response would appear as garbage input to the guest. The stdin thread discards any sequence starting with `0x1b` (ESC) until it sees a terminating letter. This is a simplified filter for ANSI escape responses generated by the host terminal - not a complete escape sequence parser.

## Execution flow

```
VMM                                      Guest (Linux + busybox)
───                                      ───────────────────────
set_raw_terminal()
pthread_create(stdin_thread)
pthread_create(vcpu_thread)
                                         kernel boots...
                                         init: open /dev/ttyS0
                                         execve("/bin/sh")
                                         sh: write prompt "~ # "
                                              ↓ (TX via THR)
stdout: "~ # "

User types 'l':
  stdin_thread: read() → 'l'
  uart_rx(): rbr='l', lsr|=DR
  KVM_IRQ_LINE(4, 1→0)
                                         IRQ 4 → serial8250_interrupt
                                         read IIR → 0x04 (RDI)
                                         read RBR → 'l'
                                         tty: echo 'l' (TX)
stdout: "l"

User types '\n':
  (same flow)
                                         sh: execute "ls"
                                         write output (TX)
stdout: "bin  dev  init  proc  sys\n"
```

## Implementation

### stdin_thread

```c
static void *stdin_thread(void *arg) {
    uint8_t c;
    while (read(STDIN_FILENO, &c, 1) == 1) {
        /* Filter ESC sequences (terminal responses) */
        if (c == 0x1b) {
            while (read(STDIN_FILENO, &c, 1) == 1) {
                if ((c >= 'A' && c <= 'Z') || (c >= 'a' && c <= 'z'))
                    break;
            }
            continue;
        }
        uart_rx(&uart, c, g_vmfd);
    }
    return NULL;
}
```

### uart_rx (in uart.c)

```c
void uart_rx(struct uart8250 *u, uint8_t c, int vmfd) {
    u->rbr = c;
    u->lsr |= UART_LSR_DR;       /* Data Ready */
    if (u->ier & UART_IER_RDI) {  /* RX interrupt enabled? */
        u->pending_rdi = 1;
        /* Raise and lower IRQ 4, creating an interrupt request
           visible to the in-kernel PIC */
        struct kvm_irq_level irq = { .irq = 4, .level = 1 };
        ioctl(vmfd, KVM_IRQ_LINE, &irq);
        irq.level = 0;
        ioctl(vmfd, KVM_IRQ_LINE, &irq);
    }
}
```

`uart_rx()` is called from the stdin thread but accesses the same UART state that the vCPU thread reads via `uart_in()`. microkvm uses the same device lock introduced in Step 9 to protect UART state.

### initramfs with busybox

```c
/* init.c — PID 1 */
int main(void) {
    mount("devtmpfs", "/dev", "devtmpfs", 0, NULL);
    mount("proc", "/proc", "proc", 0, NULL);
    mount("sysfs", "/sys", "sysfs", 0, NULL);
    close(0); close(1); close(2);
    int fd = open("/dev/ttyS0", O_RDWR);
    dup2(fd, 0); dup2(fd, 1); dup2(fd, 2);
    chdir("/root");
    char *argv[] = {"/bin/sh", NULL};
    char *envp[] = {"HOME=/root", "PATH=/bin", "PS1=~ # ", NULL};
    execve("/bin/sh", argv, envp);
}
```

The init process connects stdin/stdout/stderr to `/dev/ttyS0` (our emulated UART) and execs busybox sh.

## Building the initramfs (interactive shell)

Step 10's initramfs only printed a message. For an interactive shell, we add busybox and update init to exec `/bin/sh`:

```bash
$ mkdir -p initramfs/{dev,proc,sys,bin,root}
$ sudo mknod initramfs/dev/console c 5 1
$ sudo chown -R $(whoami) initramfs/

# Static busybox binary (provides sh, ls, cat, etc.)
$ curl -o initramfs/busybox \
  https://busybox.net/downloads/binaries/1.35.0-x86_64-linux-musl/busybox
$ chmod +x initramfs/busybox

# Create applet symlinks
$ cd initramfs/bin
$ for cmd in sh ls cat echo uname mount head ps; do
    ln -sf /busybox $cmd
done
$ cd ../..

# init: mount filesystems, connect ttyS0, exec shell
$ gcc -static -o initramfs/init init.c   # (init.c shown above)

# Pack
$ cd initramfs && find . | cpio -o -H newc | gzip > ../initramfs.gz && cd ..
```

## Key insight

### Why interactive input matters

Until Step 10, Linux was only producing output. Step 11 closes the loop by allowing input to flow back into the guest. For the first time, software running inside the VM can react to actions performed by the user outside the VM.

### The full I/O loop

Step 11 completes the **full I/O loop** for a serial device:

```
User keystroke → stdin_thread → UART RBR → IRQ 4 → Guest driver
     ↑                                                    │
     │                                                    ↓
Host terminal ← stdout ← UART THR ← Guest driver ← Shell output
```

This is a complete character device: bidirectional data flow with interrupt-driven RX notification (RDI signals data arrival to the guest).

The same pattern - data plane + interrupt notification - appears in every I/O device. The next phase of microkvm (Steps 12–14) replaces this per-character interrupt model with shared memory ring buffers (virtio), dramatically reducing the number of VM exits per I/O operation.

Typing a single character causes at least one interrupt and multiple VM exits. For large I/O workloads (e.g. `cat` on a large file), this becomes prohibitively expensive - motivating the shift to virtio in Step 12.

## What this teaches

| Concept | How it appears here |
|---------|-------------------|
| UART RX | RBR, LSR.DR, IER.RDI, IIR=0x04 |
| Separate I/O thread | stdin_thread reads host input independently of vCPU |
| IRQ as notification | IRQ 4 wakes the guest driver when data arrives |
| Raw terminal | ICANON/ECHO disabled for character-at-a-time delivery |
| ESC filtering | Discard terminal control responses |
| initramfs + init | PID 1 connects ttyS0 and execs shell |
| Full I/O loop | TX + RX + interrupts = complete character device |
