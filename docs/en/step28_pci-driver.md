# Step 28: Minimal PCI driver — probe and remove

> **Phase F: Linux PCI Driver**
>
> Phase E built the device model (VMM side). Phase F writes the driver (guest kernel side).
> Together they form both halves of a virtual PCI device: the hardware emulation and the
> software that operates it.

## Goal

Write a minimal Linux kernel module that matches the microkvm PCI device (vendor 0x1234, device 0x0001) by ID and has its `probe` function called automatically on `insmod`.

## Background

### Phase E ↔ Phase F connection

Phase E implemented the *device* — the VMM responds to config space reads with vendor=0x1234, device=0x0001. Phase F implements the *driver* — the guest kernel module that declares "I can handle devices with these IDs."

The two sides are connected by the Vendor/Device ID:
```
VMM (pci.c):     config[0x00] = 0x1234, config[0x02] = 0x0001
Driver (.c):     #define MICROKVM_VENDOR 0x1234, #define MICROKVM_DEVICE 0x0001
```

If these don't match, PCI core won't call probe. This is identical to how physical device drivers work.

### Linux PCI driver lifecycle

```
1. Boot: PCI bus scan reads vendor/device ID from each slot's config space
2. insmod: driver registers its ID table via pci_register_driver()
3. Match: PCI core finds a device with matching IDs
4. Probe: driver's probe() is called — device initialization
5. rmmod: remove() is called — device cleanup
```

### module_pci_driver macro

```c
module_pci_driver(microkvm_pci_driver);
```

Expands to `module_init` + `module_exit` + `pci_register_driver` / `pci_unregister_driver`. Eliminates boilerplate for simple PCI drivers.

### Out-of-tree module build

```makefile
obj-m := microkvm_pci.o
KDIR := $(HOME)/linux-src_v2

all:
	$(MAKE) -C $(KDIR) M=$(PWD) modules
```

`-C $(KDIR)` delegates to the kernel's Makefile; `M=$(PWD)` tells it to build modules in our directory. Requires `Module.symvers` (generated by `make modules` in kernel tree).

## Execution flow

```
Guest (insmod)               PCI Core                  VMM
──────────────               ────────                  ───
insmod microkvm_pci.ko
  → pci_register_driver()
                             walk device list
                             device 0000:00:00.0
                               vendor=0x1234 ✓
                               device=0x0001 ✓
                             → call probe()

probe():
  dev_info("probe called")
                                                       (no VMM interaction
                                                        — probe is pure kernel)

rmmod microkvm_pci
  → remove()
  → pci_unregister_driver()
```

## Implementation

### Prerequisites

- `CONFIG_MODULES=y` + `CONFIG_MODULE_UNLOAD=y` in kernel config
- `make modules` in kernel tree (generates `Module.symvers`)
- `.ko` placed in initramfs at `/lib/modules/`

### driver/microkvm_pci.c

```c
#include <linux/module.h>
#include <linux/pci.h>

/* Must match VMM's pci.h definitions */
#define MICROKVM_VENDOR  0x1234
#define MICROKVM_DEVICE  0x0001

/* Called when PCI core finds a device matching our ID table */
static int microkvm_probe(struct pci_dev *pdev, const struct pci_device_id *ent)
{
    dev_info(&pdev->dev, "microkvm_pci: probe called\n");
    return 0;
}

/* Called on rmmod or device removal */
static void microkvm_remove(struct pci_dev *pdev)
{
    dev_info(&pdev->dev, "remove called\n");
}

static const struct pci_device_id microkvm_pci_ids[] = {
    { PCI_DEVICE(MICROKVM_VENDOR, MICROKVM_DEVICE) },
    { 0, }  /* sentinel — end of table */
};
MODULE_DEVICE_TABLE(pci, microkvm_pci_ids);
```

`PCI_DEVICE()` is a macro that expands to a `struct pci_device_id` entry with the specified vendor and device IDs (other fields zeroed for wildcard matching).

```c
static struct pci_driver microkvm_pci_driver = {
    .name     = "microkvm_pci",
    .id_table = microkvm_pci_ids,
    .probe    = microkvm_probe,
    .remove   = microkvm_remove,
};
module_pci_driver(microkvm_pci_driver);

MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("microkvm PCI device driver");
```

### driver/Makefile

```makefile
obj-m := microkvm_pci.o
KDIR := $(HOME)/linux-src_v2

all:
	$(MAKE) -C $(KDIR) M=$(PWD) modules

clean:
	$(MAKE) -C $(KDIR) M=$(PWD) clean
```

## Output

```
/ # insmod /lib/modules/microkvm_pci.ko
microkvm_pci: loading out-of-tree module taints kernel.
microkvm_pci 0000:00:00.0: microkvm_pci: probe called
/ # lsmod
Module                  Size  Used by    Tainted: G
microkvm_pci           12288  0
/ # rmmod microkvm_pci
microkvm_pci 0000:00:00.0: remove called
```

`probe called` confirms: PCI core matched our driver to the VMM device via vendor/device ID and called our initialization function.

> **Note:** At this point the driver is only *attached*. It has not enabled the device or accessed any BARs yet — that comes in Step 29.

> **Note:** "taints kernel" is expected for out-of-tree modules. It marks the kernel for debugging purposes and does not indicate an error.

## Key insight

A PCI driver is fundamentally an ID table + two callbacks. The kernel's PCI subsystem handles all the complexity of bus scanning, ID matching, and lifecycle management. The driver just declares what it can handle and provides probe/remove functions. This is the same structure used by every Linux PCI driver — from simple educational modules to production NVMe and network drivers.

## What this teaches

| Concept | How it appears here |
|---------|-------------------|
| pci_device_id table | Vendor/Device ID matching (same as physical drivers) |
| probe/remove | Device lifecycle callbacks |
| module_pci_driver | Macro eliminating init/exit boilerplate |
| Out-of-tree build | `make -C $(KDIR) M=$(PWD) modules` |
| Module.symvers | Kernel symbol table for module linking |
| ID match = binding | Driver owns device only if IDs match |
